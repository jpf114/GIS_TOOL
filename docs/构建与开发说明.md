# 构建与开发说明

## 1. 当前已验证环境

当前已实际验证通过的环境如下：

- Windows 10/11
- Visual Studio 2022
- C++17
- CMake
- 全局 `vcpkg`

建议显式设置：

```powershell
$env:VCPKG_ROOT="D:\Code\GitHubCode\vcpkg"
```

如果当前终端没有显式设置 `VCPKG_ROOT`，项目也会优先尝试从 `CMAKE_TOOLCHAIN_FILE` 推断全局 `vcpkg` 根目录。

## 2. 依赖安装策略

本项目统一复用全局 `vcpkg`，不在项目构建目录内重复安装依赖。

依赖目录：

- `VCPKG_ROOT/installed/x64-windows`

建议安装：

```powershell
vcpkg install gdal opencv4 proj gtest qtbase --triplet x64-windows
```

## 3. 当前推荐构建方式

### 3.1 仅构建 CLI 与测试

```powershell
cmake -S . -B build -G "Visual Studio 17 2022" -A x64 -DGIS_BUILD_GUI=OFF -DGIS_BUILD_TESTS=ON
cmake --build build --config Debug
ctest --test-dir build -C Debug --output-on-failure
```

### 3.2 构建 GUI、CLI 与测试

```powershell
cmake -S . -B build-gui -G "Visual Studio 17 2022" -A x64 -DGIS_BUILD_GUI=ON -DGIS_BUILD_TESTS=ON
cmake --build build-gui --config Debug
ctest --test-dir build-gui -C Debug --output-on-failure
```

## 4. 当前已修复的关键问题

以下问题已经在当前主线中修复：

- 切换为全局 `vcpkg` 依赖复用，不再使用构建目录内局部依赖安装
- 兼容 `VS2022 + C++17`
- 修复 `GDALPolygonize` 在当前 GDAL 版本下的句柄调用问题
- 修复 `gtest_discover_tests` 与 Windows Debug 运行时组合导致的测试不稳定问题
- 区分 Debug / Release 运行时 DLL 复制目录
- 补齐 `PROJ` 与 `GDAL` 数据目录的运行时初始化
- 修复 GUI 构建中 `qwindows.dll / qwindowsd.dll` 平台插件复制逻辑

## 5. 当前验证结论

当前已经实际验证：

- `GIS_BUILD_GUI=OFF` 可编译通过
- `GIS_BUILD_GUI=ON` 可编译通过
- GUI 程序 `gis-gui.exe` 可启动
- 当前自动化测试为 `51/51` 通过

## 6. 当前仍需注意的事项

### 6.1 GUI 是基础执行界面，不是完整 GIS 工作台

当前 GUI 已具备：

- 插件列表
- 动态参数表单
- 执行按钮
- 进度对话框
- 执行结果反馈

但尚未形成完整桌面 GIS 常见能力，例如：

- 地图画布
- 图层树
- 结果预览
- 工程保存与打开
- 历史任务管理

### 6.2 插件功能可用不等于全量业务已验收

虽然当前测试通过，但这并不等于所有真实业务数据场景都已经完整验收。真实数据验证仍建议按插件逐项进行。

## 7. 开发约定

- 文档统一中文
- 提交信息统一中文
- 依赖统一复用全局 `vcpkg`
- 默认构建标准为 `C++17`

## 8. 建议日常流程

1. 先确认全局 `vcpkg` 依赖完整
2. 用新的构建目录执行 CMake 配置
3. 先跑编译，再跑测试
4. 修改代码后优先回归受影响插件
5. GUI 改动后至少验证一次可启动
