## 三、为什么后续要按这个顺序做

后续排序遵循一个很简单的原则：

1. 先把已经完成的成果彻底稳住
2. 再补那些已经实现但还不够完整的模块
3. 再做真实数据验收，确认不是只在合成数据上成立
4. 最后再继续扩展高价值算法能力

这样排有几个好处：

- 风险最低，不会一边继续加功能，一边把已有能力搞散
- 每一轮产出都能沉淀成稳定资产，而不是临时堆功能
- 后面如果换人接手，不需要重新判断哪些内容可信

如果反过来直接继续大规模新增算法，最容易出现的问题就是：

- 文档和程序行为不一致
- GUI 已有入口，但参数校验缺失
- 测试只覆盖 CLI，没有覆盖 GUI
- Release 能编译，但安装后的实际运行没人确认

所以当前最合理的策略仍然是：先收口，再扩展。

## 第三阶段：真实数据验收补强

### 目标

把“合成数据能过”推进到“关键功能在真实数据上也具备说服力”。

### 这个阶段不是做什么

不是优先新增算法，也不是做界面美化，而是做更接近真实使用场景的回归与验收。

### 优先顺序

1. `matching`
2. `classification.feature_stats`
3. `processing.pansharpen`
4. `spindex / ndvi`
5. `terrain`

### 为什么这样排

- `matching` 和 `classification.feature_stats` 更容易受真实输入质量影响
- `pansharpen` 和 `spindex` 属于用户容易直接比对结果质量的功能
- `terrain` 虽然重要，但当前先把结构性完整性收口更划算

### 这一阶段要做什么

1. 选择固定真实样例数据
2. 建立 quick / full 两类回归入口
3. 让 CLI 输出、关键结果文件、必要元数据都可检查
4. 对明显容易误用的参数补一轮针对性验证

### 这一阶段完成标准

- 有一批固定真实样例可重复运行
- 至少关键模块的 quick 真实数据回归稳定
- 不是只有单测通过，而是能说明真实场景下结果可信
