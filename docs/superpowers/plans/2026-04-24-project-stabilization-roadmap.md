# GIS Tool 项目稳定化与迭代路线图

**日期：** 2026-04-24

**目标：** 基于当前已有的插件式 GIS 工具原型，优先补齐“可构建、可测试、可协作”的工程基础，再逐步提升框架稳定性、插件可维护性和 GUI/发布能力。

**当前判断：**
- 项目主架构已基本成型：`core + framework + plugins + cli + gui + tests`
- 功能层面已具备一定完成度，不再是纯骨架工程
- 当前最大风险不在“缺功能”，而在“默认构建链路不稳定、测试不可移植、插件生命周期与大文件维护成本偏高”

---

## 一、项目现状摘要

### 1. 已完成的基础能力

- 已建立插件式架构，核心接口清晰，`CLI` 与 `GUI` 都围绕 `ParamSpec` 驱动
- 已集成 `GDAL / OpenCV / PROJ / Qt / GTest`
- 已存在 6 类主要插件：
  - `projection`
  - `cutting`
  - `matching`
  - `processing`
  - `vector`
  - `utility`
- 已编写基础测试与部分插件执行测试
- 最近提交记录显示项目仍在持续推进，方向一致

### 2. 当前主要问题

#### P0 级问题

- 默认 CMake 配置链路在当前机器上无法直接通过，首先卡在 `GDAL` 查找失败
- 顶层 `CMakeLists.txt` 中 `CMAKE_TOOLCHAIN_FILE` 的设置时机偏晚，存在 vcpkg 集成失效风险
- 测试代码中存在写死的本地绝对路径，无法保证换机器、换目录、CI 环境可运行
- 缺少面向协作的基础文档与 CI 验证闭环

#### P1 级问题

- `PluginManager` 在卸载时未真正调用插件销毁函数，生命周期管理存在隐患
- 大型插件文件已明显膨胀，后续继续迭代会提高回归风险和维护成本
- 插件测试覆盖不均衡，部分核心插件缺少足够的执行级验证

#### P2 级问题

- GUI 已具备基本框架，但参数校验、错误提示、执行反馈与结果浏览还偏原型阶段
- 发布打包、样例数据、用户使用指南仍不完整

---

## 二、优先级排序

### 优先级 1：先打通工程基本盘

**目的：** 让项目在新机器和协作环境中可配置、可编译、可测试。

**原因：**
- 这是所有后续工作的前置条件
- 当前最大阻塞不是算法能力，而是工程可复现性
- 若不先解决，后续每加一个功能都会放大调试和协作成本

### 优先级 2：再修框架稳定性

**目的：** 提高插件架构的可靠性，降低运行时异常和后续维护风险。

**原因：**
- 插件机制是项目核心能力
- 当前已进入“功能越来越多”的阶段，框架层不稳会影响全部插件

### 优先级 3：然后处理可维护性与测试覆盖

**目的：** 避免大文件持续膨胀，补齐核心功能的可信度。

**原因：**
- 现在的代码规模已经到了需要主动控复杂度的阶段
- 如果继续在大文件里堆功能，后续重构成本会快速上升

### 优先级 4：最后推进 GUI 体验与发布能力

**目的：** 把当前原型打磨成更接近可交付产品的工具。

**原因：**
- GUI 和打包属于重要但非首阻塞项
- 在底层构建与框架未稳定前，UI 打磨收益有限

---

## 三、分阶段工作计划

## Phase 1：构建与测试基线修复

**优先级：** `P0`

**阶段目标：**
- 让项目在标准环境下可完成 `configure + build + test`
- 清除本地路径依赖与环境偶然性

**关键任务：**

### 任务 1.1：修复 vcpkg 集成方式

- 调整顶层 `CMakeLists.txt` 的 toolchain 使用方式
- 明确项目推荐的配置命令
- 避免仅依赖用户手工设置环境后“碰巧能过”

**涉及文件：**
- `CMakeLists.txt`
- 可新增 `README.md`
- 可新增 `docs/build.md`

**验收标准：**
- 在一台未手工定制过项目路径的新环境中，可以根据文档完成配置
- 能明确区分“依赖未安装”和“脚本配置有误”两类错误

### 任务 1.2：去除测试中的绝对路径

- 将测试输出目录改为基于当前构建目录或临时目录生成
- 将插件目录改为从测试可执行文件位置推导
- 将 `PROJ_DATA` 相关逻辑从硬编码路径改为环境探测或 CMake 注入

**涉及文件：**
- `tests/test_core.cpp`
- `tests/test_plugins.cpp`
- `tests/CMakeLists.txt`

**验收标准：**
- 测试不再依赖 `D:/Develop/...` 这类绝对路径
- 同一套测试可在不同工作目录运行

### 任务 1.3：建立基础 CI

- 新增最小 CI 流程，至少覆盖：
  - CMake configure
  - build
  - test
- 优先先做 Windows 流程

**涉及文件：**
- `.github/workflows/*.yml` 或等效 CI 配置

**验收标准：**
- 每次提交都能验证构建链路是否被破坏

### 任务 1.4：补充基础使用文档

- 增加项目简介
- 增加依赖准备说明
- 增加 CLI/GUI 构建与运行说明
- 增加插件目录结构说明

**涉及文件：**
- `README.md`
- `docs/`

**验收标准：**
- 新成员拿到仓库后，不需要口头说明即可完成第一次构建尝试

**本阶段交付物：**
- 可配置、可编译、可测试的主干工程
- 最小可用协作文档
- 最小 CI 流程

---

## Phase 2：插件框架稳定性加固

**优先级：** `P0/P1`

**阶段目标：**
- 提高插件加载、执行、卸载的可靠性
- 把框架层风险收敛在功能扩展前

**关键任务：**

### 任务 2.1：修复插件销毁与卸载逻辑

- 在 `PluginManager` 中保存并调用真实的销毁函数
- 明确插件实例与动态库句柄的生命周期
- 补充异常和空指针路径处理

**涉及文件：**
- `src/framework/include/gis/framework/plugin_manager.h`
- `src/framework/src/plugin_manager.cpp`

**验收标准：**
- 插件卸载流程不依赖“直接丢弃指针”
- 生命周期逻辑可通过测试验证

### 任务 2.2：增强插件加载错误处理

- 对以下场景提供更明确的行为与错误信息：
  - 动态库无法加载
  - 缺少 `createPlugin`
  - 缺少 `destroyPlugin`
  - 插件实例创建失败
  - 重复名称插件冲突

**涉及文件：**
- `src/framework/src/plugin_manager.cpp`
- `tests/test_framework.cpp`
- 可新增专门的插件管理测试文件

**验收标准：**
- 框架在坏插件或不完整插件存在时仍能稳定运行

### 任务 2.3：统一 CLI/GUI 对参数与结果的处理约定

- 校验 `required/default/enum/extent/crs` 的行为是否一致
- 明确 action 参数在 CLI 与 GUI 中的表现
- 规范错误返回格式和消息粒度

**涉及文件：**
- `src/cli/main.cpp`
- `src/cli/cli_parser.cpp`
- `src/gui/param_widget.cpp`
- `src/framework/include/gis/framework/param_spec.h`
- `src/framework/include/gis/framework/result.h`

**验收标准：**
- 同一插件在 CLI 与 GUI 的基本行为一致
- 用户输入错误时能得到可理解反馈

**本阶段交付物：**
- 更可靠的插件管理器
- 更稳定的加载/执行/卸载流程
- 更统一的接口约定

---

## Phase 3：插件实现拆分与测试补强

**优先级：** `P1`

**阶段目标：**
- 控制插件复杂度
- 提升核心功能的可信度

**关键任务：**

### 任务 3.1：拆分大插件文件

优先拆分以下文件：
- `src/plugins/processing/processing_plugin.cpp`
- `src/plugins/vector/vector_plugin.cpp`
- `src/plugins/matching/matching_plugin.cpp`

**建议拆分方式：**
- 保留一个薄入口文件
- 按 `action` 或功能域拆成多个实现文件
- 把公用读写、校验、结果组装逻辑抽到局部工具文件

**验收标准：**
- 单文件复杂度明显下降
- 新增 action 不再需要频繁修改超大文件

### 任务 3.2：补齐核心插件执行级测试

优先补：
- `projection`
- `cutting`
- `matching`

重点测试：
- 正常路径
- 缺参场景
- 非法文件输入
- 输出元数据正确性
- 空结果或边界数据场景

**涉及文件：**
- `tests/test_plugins.cpp`
- 可拆分为多个插件测试文件

**验收标准：**
- 每个核心插件至少有一组真实执行测试，不只停留在参数定义检查

### 任务 3.3：补充回归测试基线

- 为近期新增功能补回归测试：
  - `hough`
  - `watershed`
  - `kmeans`
  - `NDVI`
  - 向量转换与 dissolve

**验收标准：**
- 后续重构插件文件时，关键能力有测试兜底

**本阶段交付物：**
- 更清晰的插件代码结构
- 更均衡的测试覆盖
- 更低的回归风险

---

## Phase 4：GUI 体验增强

**优先级：** `P2`

**阶段目标：**
- 将现有 GUI 从“可操作原型”提升为“可持续使用的桌面工具”

**关键任务：**

### 任务 4.1：补参数校验与错误提示

- 必填项校验
- 文件存在性校验
- 输出路径合理性校验
- 类型错误提示

### 任务 4.2：增强执行反馈

- 更清晰的进度显示
- 日志消息展示
- 取消执行能力
- 完成后结果路径展示与快速打开

### 任务 4.3：优化参数面板体验

- 对复杂参数分组
- 为 CRS、Extent、Enum 提供更友好的编辑方式
- 处理参数过多时的可读性问题

### 任务 4.4：统一编码与中文显示质量

- 清理当前代码中疑似编码不一致导致的中文字符串异常
- 确保 CLI、GUI、文档中的中文内容显示一致

**涉及文件：**
- `src/gui/mainwindow.cpp`
- `src/gui/param_widget.cpp`
- `src/gui/progress_dialog.cpp`
- `src/gui/crs_dialog.cpp`

**验收标准：**
- GUI 在常见插件操作下可稳定完成任务
- 用户无需阅读代码即可理解当前执行状态

**本阶段交付物：**
- 可用性更高的桌面端界面
- 更好的错误提示与任务反馈

---

## Phase 5：发布与产品化收尾

**优先级：** `P2`

**阶段目标：**
- 建立项目对外分发能力
- 让工具具备面向测试用户或内部团队交付的条件

**关键任务：**

### 任务 5.1：整理发布结构

- 明确二进制目录结构
- 确保插件与运行时依赖打包完整
- 明确 CLI 与 GUI 的最终分发内容

### 任务 5.2：增加示例数据与演示命令

- 提供最小样例数据
- 提供各主要插件的示例命令
- 给出常见输入输出样例

### 任务 5.3：整理版本说明与变更记录

- 为每次发布形成变更摘要
- 区分实验功能与稳定功能

**验收标准：**
- 非开发人员也能按说明包运行基础功能
- 发布包结构可复用，不依赖临时手工整理

**本阶段交付物：**
- 初步可分发版本
- 样例与使用指南

---

## 四、建议的执行顺序

建议严格按下面顺序推进：

1. Phase 1：构建与测试基线修复
2. Phase 2：插件框架稳定性加固
3. Phase 3：插件实现拆分与测试补强
4. Phase 4：GUI 体验增强
5. Phase 5：发布与产品化收尾

---

## 五、建议的近期两周冲刺目标

如果只看最近一轮迭代，建议先集中做下面 6 件事：

1. 修复 `CMake + vcpkg` 默认配置链路
2. 去掉测试中的绝对路径与本地环境假设
3. 新增最小 `README` 与构建说明
4. 建立最小 Windows CI
5. 修复 `PluginManager` 的插件销毁逻辑
6. 为 `projection / cutting / matching` 补首批执行级测试

这 6 件事做完后，项目就会从“作者驱动原型”提升为“可稳定协作开发的工程”。

---

## 六、资源投入建议

### 如果只有 1 人推进

- 第 1 周：只做 Phase 1
- 第 2 周：做 Phase 2，并开始补核心测试
- 第 3 周起：再进入插件拆分

### 如果有 2 人并行推进

- 方向 A：构建链路、测试环境、CI、README
- 方向 B：PluginManager 修复、插件测试补强

### 如果有 3 人并行推进

- 方向 A：工程化与 CI
- 方向 B：框架稳定性
- 方向 C：插件测试与大文件拆分准备

---

## 七、风险提醒

- 如果继续优先叠加新算法功能，短期看进度会快，长期会明显放大构建、调试和回归成本
- 如果不尽快移除测试中的环境依赖，CI 和团队协作会长期受阻
- 如果不尽快处理插件生命周期与超大文件问题，后续 GUI 接入和发布阶段会暴露更多隐性问题

---

## 八、结论

当前项目最合理的策略不是“继续扩功能”，而是先做一次工程稳定化。

**一句话排序：**
先修构建与测试基线，再稳插件框架，再补测试与拆分大插件，最后再打磨 GUI 与发布。
